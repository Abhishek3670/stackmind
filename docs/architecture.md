### Runtime Engine

The engine is the STACKMIND package itself. It provides:

- **CLI commands**: `init`, `validate`, `doctor`, `migrate`
- **Schemas**: JSON Schema definitions for all YAML files
- **Templates**: Starter files for new runtimes
- **Validators**: Four-layer validation logic
- **Migrations**: Version upgrade scripts

### Runtime Instance

Each project using STACKMIND has its own runtime instance at `.sync/`:

- **Agent state**: Boot snapshots, session counts, task assignments
- **Communication**: Inbox messages, outbox reports
- **Task tracking**: Work orders, phase progress
- **Decision history**: All architectural decisions
- **Compliance state**: Agent compliance status

## Authority Model

```
CEO (Top Manager)
    вҶ“ Product scope, priorities, releases
Claude (Senior Architect)
    вҶ“ Architecture, planning, work orders
Gemma (QA Lead)
    вҶ“ Quality gates, approvals
Workers (Codex, Gemini, Local-LLM)
    вҶ’ Implementation only
```

### Role Ownership

| Role | Owns |
|------|------|
| CEO | Product scope, priorities, releases |
| Claude | Architecture, planning, work orders, runtime state |
| Gemma | Quality gates, code review, approvals |
| Codex | Backend implementation |
| Gemini | Frontend implementation |
| Local-LLM | GitOps, CI/CD, releases |

## Key Invariants

1. All runtime state lives under `.sync/`
2. `project-root/AGENTS.md` is authoritative
3. STACKMIND never assumes any specific project
4. Templates contain zero operational history
5. Only Claude writes to `TREE.yaml` and `runtime/boot/`
6. Workers write drafts to `runtime/drafts/`, never to `runtime/boot/`

## Two Repositories

Every project has two git repositories:

1. **Project repo** (`my-project/.git`) вҖ” Code, docs, configuration
2. **Sync repo** (`my-project/.sync/.git`) вҖ” Runtime state, messages, decisions

This separation ensures:
- Runtime history is independent of code history
- Agents can be managed separately
- Compliance and audit trails are preserved
